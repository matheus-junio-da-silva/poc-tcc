# 🔍 Slither — Guia Completo de Comandos

> **Slither** é um framework de análise estática para smart contracts Solidity e Vyper, desenvolvido pela Trail of Bits / Crytic. Ele detecta vulnerabilidades, exibe informações visuais sobre contratos e fornece uma API para análises customizadas.
>
> Fonte oficial: [github.com/crytic/slither](https://github.com/crytic/slither)

## Instalação

```bash
# Usando uv (recomendado — mais rápido)
uv tool install slither-analyzer

# Usando pip
pip3 install slither-analyzer

# Usando Homebrew (macOS)
brew install slither-analyzer

# Usando Docker
docker pull trailofbits/eth-security-toolbox
```

> **Requisito:** Python 3.10+ e `solc` (compilador Solidity). Use [`solc-select`](https://github.com/crytic/solc-select) para alternar entre versões do compilador.

## Comandos Básicos de Execução

### Analisar projeto com framework (Hardhat / Foundry / Brownie / Truffle)

```bash
slither .
```

Executa Slither no diretório atual. Esta é a opção preferida quando o projeto tem dependências, pois o Slither usa o framework subjacente para compilar o código-fonte.

---

### Analisar arquivo Solidity único

```bash
slither arquivo.sol
```

Analisa um arquivo `.sol` isolado que não importa dependências externas.

---

### Analisar contrato via Etherscan

```bash
slither 0x7F37f78cBD74481E593F9C737776F7113d76B315
```

Baixa e analisa o código-fonte de um contrato publicado no Etherscan (recomenda-se ter o `solc-select` instalado para troca automática de versão do compilador).

---

### Analisar arquivo AST gerado pelo solc

```bash
slither arquivo.ast.json
```

Analisa um arquivo AST (Abstract Syntax Tree) gerado diretamente pelo compilador `solc`.

---

### Analisar projeto Embark (primeira execução)

```bash
slither . --embark-overwrite-config
```

Na primeira execução em projetos Embark, instala o plugin `@trailofbits/embark-contract-plugin` e o adiciona ao `embark.json`. Nas execuções seguintes, basta usar `slither .`.

## Opções Gerais da CLI

| Flag | Descrição |
|------|-----------|
| `--version` | Exibe a versão instalada do Slither. |
| `-h`, `--help` | Exibe a ajuda com todos os parâmetros disponíveis. |
| `--solc-disable-warnings` | Suprime os avisos do compilador `solc` na saída. |
| `--disable-color` | Remove a colorização ANSI da saída no terminal. |
| `--config-file FILE` | Usa um arquivo de configuração JSON específico (padrão: `slither.config.json`). |
| `--solc-remaps REMAPS` | Passa remapeamentos de importação ao compilador (ex: `OpenZeppelin=/path/to/oz`). |
| `--solc-args ARGS` | Passa argumentos adicionais diretamente ao compilador `solc`. |
| `--generate-patches` | Gera patches automáticos para as vulnerabilidades detectadas. |
| `--skip-assembly` | Pula a análise de blocos `assembly` inline. |

## Seleção de Detectores

Por padrão, **todos os detectores são executados**. As flags abaixo permitem personalizar a seleção.

### Listar detectores disponíveis

```bash
slither --list-detectors
```

Exibe todos os detectores disponíveis com nome, impacto e confiança.

---

### Executar apenas detectores específicos

```bash
slither arquivo.sol --detect arbitrary-send-eth,reentrancy-eth
```

Executa somente os detectores informados, separados por vírgula.

---

### Excluir detectores específicos

```bash
slither arquivo.sol --exclude naming-convention,unused-state,suicidal
```

Exclui os detectores listados da execução.

---

### Excluir por nível de severidade

```bash
# Excluir achados informativos
slither arquivo.sol --exclude-informational

# Excluir achados de otimização
slither arquivo.sol --exclude-optimization

# Excluir achados de baixa severidade
slither arquivo.sol --exclude-low

# Excluir achados de média severidade
slither arquivo.sol --exclude-medium

# Excluir achados de alta severidade
slither arquivo.sol --exclude-high
```

---

### Excluir resultados de dependências

```bash
slither . --exclude-dependencies
```

Ignora achados originados em arquivos de dependências externas (ex: `node_modules`).

---

### Definir nível de falha (exit code)

```bash
slither arquivo.sol --fail-on high
```

Define o nível mínimo de severidade que fará o processo retornar código de saída diferente de zero. Útil em pipelines de CI. Opções: `pedantic`, `low`, `medium`, `high`, `none`.

---

## Seleção de Printers

Printers exibem informações estruturadas sobre os contratos (sem detecção de vulnerabilidades). Por padrão, **nenhum printer é executado**.

### Listar printers disponíveis

```bash
slither --list-printers
```

---

### Executar um ou mais printers

```bash
slither arquivo.sol --print inheritance-graph

# Múltiplos printers separados por vírgula
slither arquivo.sol --print call-graph,function-summary,vars-and-auth
```

## Filtragem de Caminhos

### Excluir caminhos da análise

```bash
slither . --filter-paths "openzeppelin"
```

Exclui todos os resultados que estejam relacionados apenas aos arquivos cujo caminho contenha a string informada. Aceita strings literais ou expressões regulares Python.

```bash
# Excluir múltiplos arquivos com regex
slither . --filter-paths "Migrations.sol|ConvertLib.sol"
```

### Incluir apenas caminhos específicos

```bash
slither . --include-paths "contracts/core"
```

Restringe a análise aos arquivos dentro dos caminhos informados.

## Modo Triage

O modo triage permite filtrar interativamente resultados que já foram revisados, salvando as escolhas em `slither.db.json`.

```bash
slither . --triage-mode
```

Durante a execução, para cada achado o Slither pergunta se ele deve ser ocultado nas próximas execuções. Digite o número do achado ou `All` para ocultar tudo.

Para **reexibir resultados ocultos**, delete o arquivo:

```bash
rm slither.db.json
```

### Exibir achados ignorados

```bash
slither . --show-ignored-findings
```

Mostra resultados que foram marcados como ignorados no triage.

## Saída e Relatórios

### Exportar resultado para JSON

```bash
slither arquivo.sol --json resultado.json

# Exportar para stdout
slither arquivo.sol --json -
```

---

### Exportar em formato SARIF (para IDEs e GitHub Code Scanning)

```bash
slither arquivo.sol --sarif resultado.sarif
```

---

### Gerar relatório em Markdown (checklist)

```bash
slither . --checklist
```

Gera um relatório no formato Markdown com checkboxes para cada achado.

---

### Gerar Markdown com link para código-fonte no GitHub

```bash
slither . --checklist --markdown-root https://github.com/ORG/REPO/blob/COMMIT/
```

Inclui links clicáveis para as linhas de código correspondentes no repositório GitHub.

---

### Compactar saída JSON

```bash
slither . --zip saida.zip
slither . --zip saida.zip --zip-type lzma
```

Exporta os resultados JSON compactados. Tipos suportados: `lzma` (padrão), `gz`, `bz2`, `zstd`.

## Lista Completa de Detectores

Os detectores são organizados por **impacto** (High, Medium, Low, Informational, Optimization) e **confiança** (High, Medium, Low).

### 🔴 Impacto: High / Confiança: High

| Detector | O que detecta |
|----------|---------------|
| `abiencoderv2-array` | Arrays de storage com ABIEncoderV2 — bug do compilador que pode causar corrupção de dados. |
| `arbitrary-send-erc20` | `transferFrom` com parâmetro `from` controlável pelo chamador (roubo de tokens). |
| `array-by-reference` | Modificação de array de storage por valor em vez de por referência (sem efeito). |
| `encode-packed-collision` | Colisão de hash em `abi.encodePacked` com tipos de comprimento variável. |
| `incorrect-shift` | Ordem incorreta de parâmetros em instruções de shift em assembly. |
| `multiple-constructors` | Múltiplos esquemas de construtor (herança de Solidity antigo e novo). |
| `name-reused` | Nome de contrato reutilizado entre arquivos diferentes. |
| `protected-vars` | Variáveis que deveriam ser protegidas por controle de acesso mas não estão. |
| `public-mappings-nested` | Mappings públicos com variáveis aninhadas (bug do compilador). |
| `rtlo` | Presença do caractere de controle Right-To-Left Override (ataque visual). |
| `shadowing-state` | Variáveis de estado que sobrescrevem variáveis do contrato pai. |
| `suicidal` | Funções que permitem qualquer pessoa destruir o contrato (`selfdestruct`). |
| `uninitialized-state` | Variáveis de estado utilizadas sem inicialização. |
| `uninitialized-storage` | Ponteiros de storage não inicializados (perigoso em versões antigas do Solidity). |
| `unprotected-upgrade` | Contrato atualizável sem controle de acesso na função de upgrade. |

---

### 🔴 Impacto: High / Confiança: Medium

| Detector | O que detecta |
|----------|---------------|
| `arbitrary-send-erc20-permit` | `transferFrom` com `from` arbitrário usado junto com `permit`. |
| `arbitrary-send-eth` | Funções que enviam ETH para destinos controlados pelo chamador. |
| `controlled-array-length` | Atribuição de tamanho de array controlada por entrada externa. |
| `controlled-delegatecall` | Destino de `delegatecall` controlável pelo chamador. |
| `delegatecall-loop` | Funções `payable` que usam `delegatecall` dentro de um loop. |
| `incorrect-exp` | Exponenciação incorreta (confusão entre `^` bitwise XOR e `**`). |
| `incorrect-return` | Uso incorreto de `return` em modo assembly (deveria ser `leave`). |
| `msg-value-loop` | Uso de `msg.value` dentro de um loop (pode ser reutilizado indevidamente). |
| `reentrancy-eth` | Reentrância que pode resultar em roubo de Ether. |
| `reentrancy-balance` | Reentrância que leva a verificações de saldo desatualizadas. |
| `return-leave` | Uso de `return` no lugar de `leave` em assembly Yul. |
| `storage-array` | Bug do compilador em arrays de inteiros com sinal no storage. |
| `unchecked-transfer` | Transferências de tokens sem verificação do valor de retorno. |
| `weak-prng` | Geração de números pseudoaleatórios fraca (baseada em variáveis de bloco). |

---

| Detector | O que detecta |
|----------|---------------|
| `domain-separator-collision` | Token ERC20 com função cuja assinatura colide com `DOMAIN_SEPARATOR()` do EIP-2612. |
| `enum-conversion` | Conversão perigosa de enum que pode resultar em valor inválido. |
| `erc20-interface` | Interface ERC20 implementada incorretamente. |
| `erc721-interface` | Interface ERC721 implementada incorretamente. |
| `incorrect-equality` | Uso de igualdade estrita (`==`) em contextos que deveriam usar `>=` ou `<=`. |
| `locked-ether` | Contratos que recebem Ether mas não têm função para retirá-lo. |
| `mapping-deletion` | Deleção de mapping que contém uma struct (deixa dados residuais). |
| `pyth-deprecated-functions` | Uso de funções depreciadas do oráculo Pyth. |
| `pyth-unchecked-confidence` | Nível de confiança do preço Pyth não verificado. |
| `pyth-unchecked-publishtime` | `publishTime` do preço Pyth não verificado. |
| `shadowing-abstract` | Variáveis de estado que sobrescrevem variáveis de contratos abstratos pai. |
| `tautological-compare` | Comparação de variável com ela mesma (sempre true ou false). |
| `tautology` | Tautologia ou contradição lógica em condições. |
| `write-after-write` | Escrita em variável logo após outra escrita, sem leitura intermediária (escrita inútil). |

| Detector | O que detecta |

| `variable-order` | `slither arquivo.sol --print variable-order` | Exibe a ordem de armazenamento das variáveis de estado no storage. |

| `evm` | `slither arquivo.sol --print evm` | Exibe as instruções EVM correspondentes a cada nó das funções. |
| `echidna` | `slither arquivo.sol --print echidna` | Exporta informações de guia para o fuzzer Echidna (aumento de cobertura). |

> ```bash
> sudo apt install xdot
> xdot arquivo.dot
> ```
> Para converter `.dot` para PNG ou SVG:
> ```bash
> dot arquivo.dot -Tpng -o saida.png
> dot arquivo.dot -Tsvg -o saida.svg
> ```

## Ferramentas Auxiliares (Sub-comandos)

O Slither instala automaticamente ferramentas complementares como executáveis separados.

### `slither-check-upgradeability`

```bash
slither-check-upgradeability contrato.sol NomeDoContrato
```

Verifica problemas de contratos atualizáveis baseados em `delegatecall` (padrão proxy). Detecta: colisão de variáveis de storage entre versões, variáveis não inicializadas, funções de inicialização sem proteção, etc.

---

### `slither-check-erc`

```bash
slither-check-erc contrato.sol NomeDoToken --erc ERC20
slither-check-erc contrato.sol NomeDoNFT   --erc ERC721
```

Verifica a conformidade do contrato com o padrão ERC especificado. Detecta funções ausentes, assinaturas incorretas e eventos não emitidos.

---

### `slither-prop`

```bash
slither-prop contrato.sol NomeDoContrato
```

Gera automaticamente testes unitários e propriedades de segurança para uso com Echidna (fuzzer) e Manticore (execução simbólica). Útil para campanhas de fuzzing direcionadas.

---

### `slither-flat`

```bash
slither-flat contrato.sol
slither-flat contrato.sol --strategy MostDerived
```

Achata (flatten) uma base de código Solidity em um único arquivo, resolvendo todas as importações. Estratégias disponíveis: `MostDerived` (padrão), `OneFile`, `LocalImport`.

---

### `slither-read-storage`

```bash
slither-read-storage contrato.sol NomeDoContrato --rpc-url https://mainnet.infura.io/v3/KEY
slither-read-storage contrato.sol NomeDoContrato --variable-name owner
```

Lê e decodifica valores de variáveis do storage de um contrato implantado, conectando-se a um nó RPC. Útil para auditorias de contratos em produção.

---

### `slither-interface`

```bash
slither-interface contrato.sol NomeDoContrato
```

Gera automaticamente a interface Solidity (arquivo `.sol` com declarações de funções e eventos) a partir de um contrato existente.

## Arquivo de Configuração

Em vez de passar todas as flags via CLI, é possível usar um arquivo JSON. Por padrão, o Slither lê `slither.config.json` na raiz do projeto.

```json
{
    "detectors_to_run": "all",
    "printers_to_run": null,
    "detectors_to_exclude": "naming-convention,pragma",
    "exclude_dependencies": true,
    "exclude_informational": false,
    "exclude_optimization": false,
    "exclude_low": false,
    "exclude_medium": false,
    "exclude_high": false,
    "fail_on": "high",
    "json": "resultado.json",
    "sarif": null,
    "disable_color": false,
    "filter_paths": "openzeppelin",
    "generate_patches": false,
    "triage_database": "slither.db.json",
    "show_ignored_findings": false
}
```

Para usar um arquivo de configuração customizado:

```bash
slither . --config-file meu-config.json
```

> **Prioridade:** flags passadas via CLI sempre têm prioridade sobre o arquivo de configuração.

## Supressão de Avisos no Código

É possível suprimir resultados específicos diretamente no código-fonte Solidity.

### Suprimir próxima linha

```solidity
// slither-disable-next-line NOME_DO_DETECTOR
uint x = variavelProblematica;
```

### Suprimir uma seção de código

```solidity
// slither-disable-start reentrancy-eth
function funcaoProblematica() external {
    // ...
}
// slither-disable-end reentrancy-eth
```

### Marcar chamada externa como não-reentrant

```solidity
/// @custom:security non-reentrant
address public vault;
```

Indica ao Slither que as chamadas externas através desta variável são protegidas contra reentrância.
