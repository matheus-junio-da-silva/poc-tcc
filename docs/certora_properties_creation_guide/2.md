## 1. O que é o Certora Prover e CVL

O **Certora Prover** é uma ferramenta de verificação formal que analisa bytecode de contratos inteligentes contra especificações escritas em **CVL (Certora Verification Language)**. Ao contrário de testes unitários, ele verifica *todos* os estados e caminhos possíveis do contrato — não apenas os casos que o desenvolvedor pensou em testar.

A CVL tem sintaxe similar ao Solidity, mas com construções adicionais para escrever propriedades formais.

**Como funciona:**
1. Você escreve o contrato Solidity
2. Você escreve o arquivo `.spec` em CVL
3. Você executa `certoraRun contrato.sol --verify Contrato:spec.spec`
4. O Prover envia o trabalho para a nuvem Certora e retorna um relatório

## 2. Estrutura de um arquivo `.spec`

Um arquivo `.spec` CVL pode conter as seguintes seções (todas opcionais, mas devem aparecer nesta ordem se presentes):

```
pragma specify <versão>;          // opcional — pragma de versão CVL
import "outro_arquivo.spec";      // opcional — importar outras specs
using ContratoB as b;             // opcional — referenciar múltiplos contratos

methods {                          // bloco de métodos
    function minhaFuncao() external returns (uint256) envfree;
}

ghost ...                          // variáveis fantasma para rastrear estado
hook ...                           // hooks para interceptar operações de storage

definition ...                     // definições reutilizáveis (expressões)
function cvlFuncao() { ... }       // funções CVL reutilizáveis

invariant nomeInvariante() ...     // invariants (propriedades sempre verdadeiras)

rule nomeRegra() { ... }           // regras (comportamentos esperados)
```

### O bloco `methods`

Declara funções do contrato para fornecer informações adicionais ao Prover:

```cvl
methods {
    // envfree: função não depende de msg.sender, block, etc.
    function owner() external returns (address) envfree;
    function hasRole(bytes32 role, address account) external returns (bool) envfree;
    function totalSupply() external returns (uint256) envfree;

    // função que depende de ambiente (msg.sender, etc.) — não precisa de envfree
    // function transfer(address to, uint256 amount) external returns (bool);
}
```

### O tipo `env`

O `env` é um tipo especial que agrupa variáveis globais do Solidity disponíveis em CVL:

```cvl
rule exemplo() {
    env e;
    // e.msg.sender  — endereço do remetente
    // e.msg.value   — valor em wei enviado
    // e.block.number — número do bloco atual
    // e.block.timestamp — timestamp do bloco atual
}
```

## 3. Conceitos fundamentais para Access Control

### `@withrevert` e `lastReverted`

Para verificar se uma função reverte sob certas condições, use `@withrevert`. Após a chamada, a variável builtin `lastReverted` indica se a chamada reverteu.

```cvl
rule somenteDono() {
    env e;
    funcaoRestrita@withrevert(e);
    // Se o chamador não for o dono, a transação DEVE reverter
    assert e.msg.sender != owner() => lastReverted;
}
```

### Implicação (`=>`) vs Bicondicional (`<=>`)

- `P => Q` — "Se P, então Q" (mais permissivo, prova uma direção)
- `P <=> Q` — "P se e somente se Q" (mais rígido, prova ambas as direções)

**Na prática para access control:**
- Use `=>` quando quiser apenas provar que não-autorizados revertem
- Use `<=>` quando quiser provar que autorizados têm sucesso E não-autorizados revertem

### Regras Paramétricas

Uma regra paramétrica itera sobre *todas* as funções do contrato:

```cvl
rule somenteDonoPodeModificar(method f) {
    env e;
    calldataarg args;
    // ... lógica que se aplica a qualquer método f
    f(e, args);
    // ...
}
```

### `filtered`

Usado para restringir quais funções uma regra paramétrica verifica:

```cvl
rule exemplo(method f) 
    filtered { f -> !f.isView && !f.isPure }
{
    // ...
}
```

### `invariant`

Um invariant declara uma propriedade que deve ser verdadeira *sempre* — antes do constructor, e antes e depois de cada função:

```cvl
invariant ownerNuncaZero()
    owner() != 0
```

## 4. Vulnerabilidades de Access Control em Solidity

As principais vulnerabilidades de access control que o Certora pode verificar:

| Vulnerabilidade | Descrição | Tipo de Propriedade CVL |
|---|---|---|
| Função sem modifier | Função privilegiada acessível por qualquer um | `rule` com `@withrevert` |
| Ownership para address(0) | Owner transferido para endereço zero | `invariant` |
| Transferência não confirmada | 2-step ownership sem etapa de aceitação | `rule` de transição de estado |
| Role admin incorreto | Admin do role pode ser qualquer um | `invariant` + `rule` |
| Inicialização sem proteção | `initialize()` pode ser chamada por qualquer um | `rule` |
| Re-inicialização | `initialize()` pode ser chamada mais de uma vez | `rule` com estado antes/depois |

### 5.1 Ownable — `onlyOwner`

**Contrato de referência (simplificado):**
```solidity
contract MeuContrato {
    address public owner;
    uint256 public valor;

    modifier onlyOwner() {
        require(msg.sender == owner, "not owner");
        _;
    }

    function setValor(uint256 _v) public onlyOwner { valor = _v; }
    function pause() public onlyOwner { /* ... */ }
}
```

**Spec CVL:**
```cvl
methods {
    function owner() external returns (address) envfree;
    function valor() external returns (uint256) envfree;
}

// PROPRIEDADE 1: Não-dono NÃO pode chamar setValor com sucesso
rule naoDonoNaoPodeSetValor(env e) {
    uint256 v;
    setValor@withrevert(e, v);
    assert e.msg.sender != owner() => lastReverted,
        "non-owner chamou setValor com sucesso";
}

// PROPRIEDADE 2: Dono PODE chamar setValor (desde que sem revert por outras razões)
// Verifica a direção oposta: dono não é bloqueado indevidamente
rule donoPodemChamarSetValor(env e) {
    uint256 v;
    require e.msg.sender == owner();
    require e.msg.value == 0; // setValor não é payable

    setValor@withrevert(e, v);
    assert !lastReverted,
        "owner foi impedido de chamar setValor";
}

// PROPRIEDADE 3: Bicondicional — setValor reverte SE E SOMENTE SE não é o dono
// (precisa eliminar outras causas de revert antes)
rule apenasDonoSetValor(env e) {
    uint256 v;
    require e.msg.value == 0;

    setValor@withrevert(e, v);
    assert !lastReverted <=> e.msg.sender == owner(),
        "access control falhou para setValor";
}

// PROPRIEDADE 4: O valor só muda se o chamador for o dono
rule valorSoMudaComDono(method f) {
    env e;
    calldataarg args;

    uint256 valorAntes = valor();
    f(e, args);
    uint256 valorDepois = valor();

    assert valorDepois != valorAntes => e.msg.sender == owner(),
        "valor mudou sem autorização do dono";
}
```

### 5.2 Transferência de Ownership

**Contrato de referência:**
```solidity
contract Ownable {
    address public owner;
    address public pendingOwner; // para 2-step transfer

    function transferOwnership(address newOwner) public onlyOwner {
        owner = newOwner;
    }

    // 2-step pattern:
    function proposeOwner(address newOwner) public onlyOwner {
        pendingOwner = newOwner;
    }
    function acceptOwnership() public {
        require(msg.sender == pendingOwner);
        owner = pendingOwner;
        pendingOwner = address(0);
    }
}
```

**Spec CVL:**
```cvl
methods {
    function owner() external returns (address) envfree;
    function pendingOwner() external returns (address) envfree;
}

// PROPRIEDADE 5: Owner nunca pode ser o endereço zero
invariant ownerNaoZero()
    owner() != 0

// PROPRIEDADE 6: Só o dono pode transferir ownership
rule soDonoTransfereOwnership(env e) {
    address novoOwner;
    transferOwnership@withrevert(e, novoOwner);
    assert e.msg.sender != owner() => lastReverted,
        "non-owner executou transferOwnership";
}

// PROPRIEDADE 7: Ownership só muda via transferOwnership
rule ownerSoMudaViaTranferencia(method f) {
    env e;
    calldataarg args;

    address ownerAntes = owner();
    f(e, args);
    address ownerDepois = owner();

    assert ownerDepois != ownerAntes =>
        f.selector == sig:transferOwnership(address).selector ||
        f.selector == sig:acceptOwnership().selector,
        "owner mudou por função não autorizada";
}

// PROPRIEDADE 8: Transferência para address(0) deve reverter
rule transferParaZeroReverte(env e) {
    transferOwnership@withrevert(e, 0);
    assert lastReverted,
        "transferência para address(0) não reverteu";
}

// PROPRIEDADE 9: 2-step — acceptOwnership só funciona para pendingOwner
rule sooPendingOwnerAceita(env e) {
    acceptOwnership@withrevert(e);
    assert e.msg.sender != pendingOwner() => lastReverted,
        "acceptOwnership executado por quem não é pendingOwner";
}

// PROPRIEDADE 10: Após acceptOwnership, o owner é quem aceitou
rule aceitacaoMudaOwner(env e) {
    require e.msg.sender == pendingOwner();
    require e.msg.value == 0;
    address pendingAntes = pendingOwner();

    acceptOwnership(e);

    assert owner() == pendingAntes,
        "owner não foi atualizado após acceptOwnership";
}
```

### 5.3 Role-Based Access Control (RBAC)

**Contrato de referência (OpenZeppelin AccessControl):**
```solidity
contract MeuContrato is AccessControl {
    bytes32 public constant MINTER_ROLE = keccak256("MINTER_ROLE");
    bytes32 public constant PAUSER_ROLE = keccak256("PAUSER_ROLE");

    function mint(address to, uint256 amount) public onlyRole(MINTER_ROLE) { ... }
    function pause() public onlyRole(PAUSER_ROLE) { ... }
}
```

**Spec CVL:**
```cvl
methods {
    function hasRole(bytes32 role, address account) external returns (bool) envfree;
    function getRoleAdmin(bytes32 role) external returns (bytes32) envfree;
    function MINTER_ROLE() external returns (bytes32) envfree;
    function PAUSER_ROLE() external returns (bytes32) envfree;
    function DEFAULT_ADMIN_ROLE() external returns (bytes32) envfree;
}

// PROPRIEDADE 11: Só quem tem MINTER_ROLE pode chamar mint
rule soMinterPodeMint(env e) {
    address to;
    uint256 amount;

    mint@withrevert(e, to, amount);
    assert !hasRole(MINTER_ROLE(), e.msg.sender) => lastReverted,
        "mint chamado por quem não tem MINTER_ROLE";
}

// PROPRIEDADE 12: Só quem tem PAUSER_ROLE pode pausar
rule soPauserPodePausar(env e) {
    pause@withrevert(e);
    assert !hasRole(PAUSER_ROLE(), e.msg.sender) => lastReverted,
        "pause chamado por quem não tem PAUSER_ROLE";
}

// PROPRIEDADE 13: Só o admin do role pode conceder o role
rule soAdminPodeGrantRole(env e) {
    bytes32 role;
    address account;

    grantRole@withrevert(e, role, account);
    bytes32 adminRole = getRoleAdmin(role);
    assert !hasRole(adminRole, e.msg.sender) => lastReverted,
        "grantRole chamado por quem não é admin do role";
}

// PROPRIEDADE 14: Só o admin do role pode revogar o role
rule soAdminPodeRevokeRole(env e) {
    bytes32 role;
    address account;

    revokeRole@withrevert(e, role, account);
    bytes32 adminRole = getRoleAdmin(role);
    assert !hasRole(adminRole, e.msg.sender) => lastReverted,
        "revokeRole chamado por quem não é admin do role";
}

// PROPRIEDADE 15: Um account só pode renunciar ao seu PRÓPRIO role
rule renounceApenasProprioRole(env e) {
    bytes32 role;
    address account;

    renounceRole@withrevert(e, role, account);
    assert e.msg.sender != account => lastReverted,
        "renounceRole chamado para outra conta";
}

// PROPRIEDADE 16: Após grantRole, a conta tem o role
rule grantRoleFunciona(env e) {
    bytes32 role;
    address account;

    bytes32 adminRole = getRoleAdmin(role);
    require hasRole(adminRole, e.msg.sender);
    require e.msg.value == 0;

    grantRole(e, role, account);

    assert hasRole(role, account),
        "conta não recebeu o role após grantRole";
}

// PROPRIEDADE 17: Após revokeRole, a conta NÃO tem mais o role
rule revokeRoleFunciona(env e) {
    bytes32 role;
    address account;

    bytes32 adminRole = getRoleAdmin(role);
    require hasRole(adminRole, e.msg.sender);
    require e.msg.value == 0;

    revokeRole(e, role, account);

    assert !hasRole(role, account),
        "conta ainda tem o role após revokeRole";
}

// PROPRIEDADE 18: Roles só mudam via grantRole, revokeRole ou renounceRole
rule rolesSoMudamViasCorretas(method f) {
    env e;
    calldataarg args;
    bytes32 role;
    address account;

    bool tinhaAntes = hasRole(role, account);
    f(e, args);
    bool temDepois = hasRole(role, account);

    assert tinhaAntes != temDepois =>
        f.selector == sig:grantRole(bytes32,address).selector   ||
        f.selector == sig:revokeRole(bytes32,address).selector  ||
        f.selector == sig:renounceRole(bytes32,address).selector,
        "role mudou por função não esperada";
}
```

### 5.4 Parametric rules — qualquer função privilegiada

Este padrão é poderoso: verifica que *qualquer* função que modifica estado sensível requer autorização.

```cvl
methods {
    function owner() external returns (address) envfree;
    function paused() external returns (bool) envfree;
}

// PROPRIEDADE 19: Nenhuma função pode mudar o estado de paused sem ser o dono
// (adapte para o seu contrato — aqui supondo que só dono pode pausar/despausar)
rule pausedSoMudaComDono(method f) {
    env e;
    calldataarg args;

    bool pausedAntes = paused();
    f(e, args);
    bool pausedDepois = paused();

    assert pausedDepois != pausedAntes => e.msg.sender == owner(),
        "estado paused mudou sem ser o owner";
}

// PROPRIEDADE 20: Funções de escrita (não-view) só são bem-sucedidas para autorizados
// Exemplo: qualquer função que não seja view deve reverter para address(0) como sender
rule enderecoZeroNaoPodeChamarFuncoes(method f)
    filtered { f -> !f.isView }
{
    env e;
    calldataarg args;
    require e.msg.sender == 0;

    f@withrevert(e, args);
    assert lastReverted,
        "address(0) conseguiu chamar função sem reverter";
}

// PROPRIEDADE 21: Funções view não modificam storage (sanidade)
rule viewNaoModificaStorage(method f)
    filtered { f -> f.isView }
{
    env e;
    calldataarg args;

    // Captura estado relevante antes
    address ownerAntes = owner();
    bool pausedAntes = paused();

    f(e, args);

    // Estado não deve ter mudado
    assert owner() == ownerAntes && paused() == pausedAntes,
        "função view modificou storage";
}
```

### 5.5 Inicialização e constructor

```cvl
methods {
    function owner() external returns (address) envfree;
    function initialized() external returns (bool) envfree; // se existir
}

// PROPRIEDADE 22: Owner é definido no constructor e nunca é zero logo após
// (verificado via invariant, que testa o estado pós-constructor)
invariant ownerDefinidoNoConstructor()
    owner() != 0

// PROPRIEDADE 23: initialize() só pode ser chamada uma vez
// Supondo que existe uma variável _initialized ou similar
rule initializeSoUmaVez(env e) {
    require initialized(); // já foi inicializado

    initialize@withrevert(e); // tenta inicializar de novo
    assert lastReverted,
        "initialize() foi chamada mais de uma vez";
}

// PROPRIEDADE 24: Só o deployer pode chamar initialize (se houver essa restrição)
rule soDeployerPodeInicializar(env e) {
    address deployerEsperado; // fornecido como argumento havoc

    require !initialized(); // ainda não inicializado

    initialize@withrevert(e);
    // Se a função não reverteu, e.msg.sender deve ser autorizado
    // (adapte conforme a lógica do seu contrato)
    assert !lastReverted => e.msg.sender == deployerEsperado,
        "initialize chamado por endereço não autorizado";
}
```

### 5.6 Pausable e guards de estado

```cvl
methods {
    function paused() external returns (bool) envfree;
    function owner() external returns (address) envfree;
}

// PROPRIEDADE 25: Funções bloqueadas quando pausado devem reverter
rule funcaoReverteSePausado(env e) {
    require paused();

    uint256 amount;
    address to;

    transfer@withrevert(e, to, amount); // adapte ao nome da função
    assert lastReverted,
        "transfer não reverteu quando contrato está pausado";
}

// PROPRIEDADE 26: Só o dono pode pausar
rule soDonoPodemPausar(env e) {
    pause@withrevert(e);
    assert e.msg.sender != owner() => lastReverted,
        "pause chamado por não-owner";
}

// PROPRIEDADE 27: Só o dono pode despausar
rule soDonoPodemDespausar(env e) {
    unpause@withrevert(e);
    assert e.msg.sender != owner() => lastReverted,
        "unpause chamado por não-owner";
}

// PROPRIEDADE 28: Estado de paused só muda via pause/unpause
rule pausedSoMudaViaFuncoesCorretas(method f) {
    env e;
    calldataarg args;

    bool pausedAntes = paused();
    f(e, args);
    bool pausedDepois = paused();

    assert pausedDepois != pausedAntes =>
        f.selector == sig:pause().selector ||
        f.selector == sig:unpause().selector,
        "estado paused mudou por função inesperada";
}
```

### 5.7 Allowance / delegação de permissão

Aplicável a tokens ERC20 e contratos que usam padrão de aprovação:

```cvl
methods {
    function allowance(address owner, address spender) external returns (uint256) envfree;
    function balanceOf(address account) external returns (uint256) envfree;
}

// PROPRIEDADE 29: Só o holder pode modificar o próprio allowance
// (via approve)
rule soHolderModificaAllowance(method f) {
    env e;
    calldataarg args;

    address owner;
    address spender;
    uint256 allowanceAntes = allowance(owner, spender);
    f(e, args);
    uint256 allowanceDepois = allowance(owner, spender);

    // Se o allowance de `owner` aumentou, só `owner` pode ter feito isso
    assert allowanceDepois > allowanceAntes => e.msg.sender == owner,
        "allowance aumentou sem autorização do owner";
}

// PROPRIEDADE 30: transferFrom só reduz allowance se o caller tiver allowance suficiente
rule transferFromRequerAllowance(env e) {
    address from;
    address to;
    uint256 amount;

    uint256 allowanceAntes = allowance(from, e.msg.sender);
    transferFrom@withrevert(e, from, to, amount);

    assert !lastReverted => allowanceAntes >= amount,
        "transferFrom executada sem allowance suficiente";
}
```

### 5.8 Ghosts e Hooks para rastrear mudanças de papel

Ghosts permitem rastrear informações que não estão diretamente disponíveis via chamadas de função — por exemplo, quantas vezes um papel foi concedido, ou quem concedeu um papel.

```cvl
// Ghost para rastrear se um role foi concedido (true = foi concedido alguma vez)
ghost mapping(bytes32 => mapping(address => bool)) roleGhostConcedido {
    init_state axiom forall bytes32 r. forall address a. !roleGhostConcedido[r][a];
}

// Hook: intercepta escritas no storage de _roles (estrutura interna do AccessControl OZ)
// ATENÇÃO: o slot exato depende do contrato — adapte conforme o layout de storage
hook Sstore _roles[KEY bytes32 role].(offset 0)[KEY address account] bool value STORAGE {
    roleGhostConcedido[role][account] = value;
}

// PROPRIEDADE 31: Se o ghost diz que o role foi concedido, hasRole deve confirmar
invariant ghostConsistenteComHasRole(bytes32 role, address account)
    roleGhostConcedido[role][account] == hasRole(role, account)

// Ghost para contar quantas contas têm um determinado role
ghost mapping(bytes32 => mathint) contadorRole {
    init_state axiom forall bytes32 r. contadorRole[r] == 0;
}

hook Sstore _roles[KEY bytes32 role].(offset 0)[KEY address account] bool newValue STORAGE {
    bool oldValue;
    // Atualiza contador
    if (newValue && !oldValue) {
        contadorRole[role] = contadorRole[role] + 1;
    } else if (!newValue && oldValue) {
        contadorRole[role] = contadorRole[role] - 1;
    }
}

// PROPRIEDADE 32: Número de detentores de um role nunca é negativo
invariant contadorRoleNaoNegativo(bytes32 role)
    contadorRole[role] >= 0
```

> **Aviso:** A sintaxe exata de hooks depende do layout de storage do contrato. O exemplo acima é ilustrativo. Consulte a [documentação oficial de hooks](https://docs.certora.com/en/latest/docs/cvl/hooks.html) e inspecione o layout de storage do seu contrato específico com `certoraRun --assert_autofinder_success`.

### 5.9 Invariants de access control

Invariants são propriedades que devem valer em *qualquer* estado do contrato — após o constructor e após qualquer chamada de função.

```cvl
methods {
    function owner() external returns (address) envfree;
    function hasRole(bytes32 role, address account) external returns (bool) envfree;
    function DEFAULT_ADMIN_ROLE() external returns (bytes32) envfree;
}

// PROPRIEDADE 33: Owner nunca é address(0)
invariant ownerNuncaZero()
    owner() != 0

// PROPRIEDADE 34: Endereço zero nunca tem nenhum role
invariant enderecoZeroSemRole(bytes32 role)
    !hasRole(role, 0)

// PROPRIEDADE 35: O contrato em si nunca é owner (evita self-admin problemático)
// (opcional — depende do design)
invariant contratoNaoEOwner()
    owner() != currentContract

// PROPRIEDADE 36: Preserved block — garante que o invariant é preservado mesmo
// quando funções recebem endereços arbitrários
invariant ownerNuncaZeroComPreserved()
    owner() != 0
{
    preserved transferOwnership(address newOwner) with (env e) {
        // Para esta função, exigimos explicitamente que o novo dono não seja zero
        require newOwner != 0;
    }
    preserved proposeOwner(address newOwner) with (env e) {
        require newOwner != 0;
    }
}
```

## 6. Arquivo `.conf` de exemplo

O arquivo de configuração `.conf` (JSON) define como executar o Certora Prover:

```json
{
    "files": [
```

## 7. Erros comuns e como evitá-los

### 1. Esquecer `e.msg.value == 0` para funções não-payable

Funções não-payable revertem quando `msg.value != 0`. Se você usa `<=>` e não adiciona esse precondition, o Prover encontrará esse caso como contraexemplo.

```cvl
// ERRADO — vai falhar porque msg.value != 0 também causa revert
rule apenasDonoSetValor(env e) {
    setValor@withrevert(e, 42);
    assert !lastReverted <=> e.msg.sender == owner();
}

// CORRETO
rule apenasDonoSetValor(env e) {
    require e.msg.value == 0;
    setValor@withrevert(e, 42);
    assert !lastReverted <=> e.msg.sender == owner();
}
```

### 2. Confundir `=>` com `<=>`

- `=>` (implicação) é assimétrico: "se não-dono => reverte" não prova que o dono tem sucesso.
- Use `<=>` quando quiser provar ambas as direções.

### 3. Variáveis de método fora do corpo principal da regra (CVL 5+)

A partir do certora-cli 5.0, variáveis do tipo `method` só podem ser declaradas no corpo principal da regra (não dentro de `if`, `preserved`, etc.).

```cvl
// ERRADO (CVL 5+)
rule exemplo() {
    if (cond) {
        method f; // erro!
    }
}

// CORRETO
rule exemplo(method f) { // declara como parâmetro
    // ...
}
```

### 4. Usar `require false` em `preserved` sem entender as implicações

`require false` em um preserved block torna a verificação trivialmente verdadeira para aquela função — equivale a excluí-la da verificação. Use com cuidado e documente o motivo.

### 5. Não usar `sig:` para seletores de função (CVL 2+)

```cvl
// ERRADO (CVL 1 — não funciona em CVL 2+)
assert f.selector == transferOwnership(address).selector;

// CORRETO (CVL 2+)
assert f.selector == sig:transferOwnership(address).selector;
```
