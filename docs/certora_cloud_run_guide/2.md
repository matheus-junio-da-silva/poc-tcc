## 1. Pré-requisitos e Instalação

### Dependências obrigatórias

| Dependência | Versão mínima | Verificação |
|---|---|---|
| Python | 3.9+ | `python3 --version` |
| Java (JDK) | 21+ | `java --version` |
| Solidity compiler | v0.5+ (recomendado) | `solc --version` |

### Instalando o compilador Solidity

A forma recomendada é usar o `solc-select`:

```bash
pip install solc-select
solc-select install 0.8.23
solc-select use 0.8.23
```

Ou baixar o binário diretamente do [repositório oficial do Solidity](https://github.com/ethereum/solidity/releases) e colocar no `$PATH`.

### Instalando o certora-cli

```bash
pip3 install certora-cli
```

> **Atenção:** O terminal pode avisar que o executável `certoraRun` não está no `$PATH`. Nesse caso, adicione o caminho indicado no aviso ao seu `$PATH`.

**Linux** — adicione ao `~/.profile`:
```bash
PATH="$PATH:/home/<usuario>/.local/bin"
```
Depois execute `source ~/.profile`.

**macOS** — adicione ao `~/.zshenv`:
```bash
export PATH="$PATH:/Users/<usuario>/Library/Python/3.x/bin"
```
Depois execute `source ~/.zshenv`.

### Instalando a versão beta (opcional)

```bash
pip uninstall certora-cli
pip install certora-cli-beta
```

## 2. Configurando a Chave de Acesso

O Certora Prover requer uma chave de acesso pessoal. Você pode obter uma chave gratuita em [certora.com/signup](https://www.certora.com/signup?plan=prover).

### Configuração temporária (válida apenas no terminal atual)

```bash
export CERTORAKEY=<sua_chave_pessoal>
```

### Configuração permanente

**Linux** — adicione ao `~/.profile`:
```bash
export CERTORAKEY=<sua_chave_pessoal>
```

**macOS** — adicione ao `~/.zshenv`:
```bash
export CERTORAKEY=<sua_chave_pessoal>
```

Após editar o arquivo, execute `source ~/.profile` (Linux) ou `source ~/.zshenv` (macOS).

## 3. Executando o Prover — Comandos Básicos

### Comando principal

```bash
certoraRun <arquivo.sol>:<NomeContrato> --verify <NomeContrato>:<arquivo.spec>
```

Se o nome do arquivo `.sol` for idêntico ao nome do contrato, pode simplificar:

```bash
certoraRun <arquivo.sol> --verify <NomeContrato>:<arquivo.spec>
```

### Exemplo concreto

```bash
certoraRun contracts/Bank.sol --verify Bank:certora/specs/Bank.spec
```

### Executando via arquivo `.conf`

```bash
certoraRun meu_projeto.conf
```

### Ver todas as opções disponíveis

```bash
certoraRun --help
```

## 4. Obtendo Resultados no Terminal

**Por padrão, o `certoraRun` envia o job e encerra sem esperar o resultado.** O terminal exibe apenas o link para acompanhar no dashboard:

```
Job submitted to server

Follow your job at https://prover.certora.com
Once the job is completed, the results will be available at https://prover.certora.com/output/...
```

### Forçar o terminal a aguardar e exibir os resultados

Use o flag `--wait_for_results`:

```bash
certoraRun Bank.sol --verify Bank:Bank.spec --wait_for_results ALL
```

Com esse flag:
- O processo **fica rodando** até o job terminar na nuvem.
- O terminal imprime o resultado de cada regra quando ele fica pronto.
- O **exit code** será diferente de zero se alguma propriedade for violada (útil em scripts).

### Opções do `--wait_for_results`

| Valor | Comportamento |
|---|---|
| `ALL` (ou sem valor) | Aguarda todos os resultados e exibe no terminal |
| `NONE` | Envia o job e encerra imediatamente (exit code 0 se enviado com sucesso) |

> **Nota sobre CI:** Em ambientes de CI, o comportamento padrão **já é aguardar** os resultados e retornar exit code não-zero em caso de violação. Você pode forçar o comportamento contrário com `--wait_for_results NONE`.

### Exemplo com mensagem descritiva

Sempre adicione `--msg` para facilitar identificar o job no dashboard:

```bash
certoraRun Bank.sol \
  --verify Bank:Bank.spec \
  --msg "fix: corrigindo regra transferSpec" \
  --wait_for_results ALL
```

## 5. Verificando o Status de um Job

### Pelo link do terminal

Após enviar um job, o terminal exibe uma URL no formato:
```
https://prover.certora.com/output/<id_usuario>/<hash_job>/
```

Acesse essa URL no navegador para ver o relatório completo.

### Status via URL especial (jobStatus)

Substitua `output` por `jobStatus` na URL recebida:

```
# URL original:
https://prover.certora.com/output/65540/270dce9623d492937d82/?anonymousKey=...

# URL de status:
https://prover.certora.com/jobStatus/65540/270dce9623d492937d82/?anonymousKey=...
```

A página `jobStatus` exibe:
- Argumentos usados na linha de comando
- A mensagem (`--msg`) do job
- A versão exata do Certora Prover usada
- O **status atual** do job (se ainda está processando ou terminou)

> Essa página é criada **assim que o job é submetido**, mesmo antes de terminar.

### Dashboard principal

Acesse [prover.certora.com](https://prover.certora.com) para ver a lista de todos os seus jobs recentes, incluindo status e links.

## 6. Arquivos de Configuração (.conf)

Para projetos maiores, use arquivos `.conf` (formato JSON5) ao invés de flags no terminal.

### Executando com conf

```bash
certoraRun meu_projeto.conf
```

### Estrutura básica de um .conf

```json5
{
    "files": [
        "contracts/Bank.sol"
    ],
    "verify": "Bank:certora/specs/Bank.spec",
    "solc": "solc0.8.23",
    "msg": "verificacao completa",
    "wait_for_results": "ALL"
}
```

### Mapeamento: CLI → JSON

| CLI | .conf |
|---|---|
| `--verify Bank:Bank.spec` | `"verify": "Bank:Bank.spec"` |
| `--msg "texto"` | `"msg": "texto"` |
| `--rule nomeRegra` | `"rule": ["nomeRegra"]` |
| `--solc solc0.8.23` | `"solc": "solc0.8.23"` |
| `--loop_iter 3` | `"loop_iter": "3"` |
| `--multi_assert_check` (flag bool) | `"multi_assert_check": true` |
| `--wait_for_results ALL` | `"wait_for_results": "ALL"` |

### Gerando um .conf automaticamente

Após cada execução bem-sucedida, o `certoraRun` salva automaticamente o `.conf` usado em:

```
.certora_internal/latest/run.conf
```

Use esse arquivo como ponto de partida para criar seu próprio `.conf`.

### Exemplo com múltiplos contratos e compiladores

```json5
{
    "files": [
        "SolcArgs/A.sol",
        "SolcArgs/A.sol:B",
        "SolcArgs/C.sol"
    ],
    "verify": "A:SolcArgs/Trigger.spec",
    "solc_map": {
        "SolcArgs/A.sol": "solc6.1",
        "B": "solc6.1",
        "C": "solc5.12"
    },
    "multi_assert_check": true
}
```

## 7. Opções CLI Mais Importantes

### Seleção de regras

**Rodar apenas regras específicas** (economiza muito tempo):
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --rule transferSpec withdrawSucceeds
```

**Excluir regras específicas:**
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --exclude_rule "withdraw*"
```

**Rodar regras pesadas em jobs separados** (`split_rules`):
```bash
certoraRun Bank.sol --verify Bank:Bank.spec \
  --split_rules address_zero_cannot_become_an_account
```
> Com `split_rules`, o job original retorna imediatamente com um link para o dashboard mostrando o status dos jobs gerados.

**Filtrar por método em regras paramétricas:**
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --method 'deposit(uint256)'
```

### Controle de tempo

**Timeout global** (máximo 7200 segundos = 2 horas):
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --global_timeout 3600
```

**Timeout por regra (SMT):**
```bash
# via prover_args
certoraRun Bank.sol --verify Bank:Bank.spec --prover_args "-smt_timeout 300"
```

**Controle de concorrência** (reduz uso de memória):
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --max_concurrent_rules 2
```

### Controle do compilador Solidity

```bash
# Especificar versão do solc
certoraRun Bank.sol --verify Bank:Bank.spec --solc solc0.8.23

# Versões diferentes por contrato
certoraRun A.sol B.sol --verify A:spec.spec \
  --solc_map A.sol=solc0.8.23,B.sol=solc0.7.6
```

### Loops

```bash
# Definir número de iterações de loop
certoraRun Bank.sol --verify Bank:Bank.spec --loop_iter 3

# Tornar loop unwind condition como require (ignora casos não cobertos)
certoraRun Bank.sol --verify Bank:Bank.spec --loop_iter 3 --optimistic_loop
```

### Qualidade das especificações

**Verificar sanidade das regras** (detecta regras vacuosas/tautológicas):
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --rule_sanity basic
```

Valores para `rule_sanity`: `none`, `basic`, `advanced`.

**Múltiplos counterexamples por regra:**
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --multi_assert_check
```

**Múltiplos exemplos por resultado:**
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --multi_example
```

### Auto dispatcher (resolver chamadas externas automaticamente)

```bash
certoraRun Bank.sol --verify Bank:Bank.spec --auto_dispatcher
```

### Checagem local antes de enviar

**Apenas compilação e checagem de tipos (sem enviar para a nuvem):**
```bash
certoraRun Bank.sol --verify Bank:Bank.spec --compilation_steps_only
```
> Útil para validar a spec localmente antes de gastar créditos de verificação.

## 8. Depurando Propriedades com Problemas

### Por que o terminal não mostra erros de propriedades?

Por padrão, o job é **assíncrono**: o CLI envia o trabalho para a nuvem e encerra. Os resultados detalhados (quais regras passaram, quais falharam, counterexamples) ficam disponíveis apenas no dashboard.

### Solução: usar `--wait_for_results ALL`

```bash
certoraRun Bank.sol --verify Bank:Bank.spec --wait_for_results ALL
```

Com isso, o terminal passa a mostrar o status das regras à medida que terminam.

### Interpretando os resultados

| Status | Significado |
|---|---|
| **VERIFIED** | A propriedade foi provada para todos os caminhos possíveis |
| **VIOLATED** | Foi encontrado um counterexample (contraexemplo) que viola a regra |
| **TIMEOUT** | O solver não terminou dentro do tempo limite |
| **SANITY** | A regra falhou em checks de sanidade (ex: regra vacuosa) |
| **ERROR** | Ocorreu um erro durante a verificação |

### Regra violada: o que fazer

1. **No terminal (com `--wait_for_results`):** você verá indicação de violação e a URL do job.
2. **No dashboard:** clique na regra violada para ver o **Call Trace** e os valores das variáveis que geraram o counterexample.
3. **Isole a regra problemática:**
   ```bash
   certoraRun Bank.sol --verify Bank:Bank.spec \
     --rule nomeRegraProblem \
     --wait_for_results ALL
   ```
4. **Verifique o método específico:**
   ```bash
   certoraRun Bank.sol --verify Bank:Bank.spec \
     --rule nomeRegra \
     --method 'nomeMetodo(uint256)' \
     --wait_for_results ALL
   ```

### Regra vacuosa (sempre passa sem sentido)

Use `--rule_sanity` para detectar:

```bash
certoraRun Bank.sol --verify Bank:Bank.spec --rule_sanity basic
```

Uma regra vacuosa significa que as condições `require` são tão restritivas que nunca há um estado de entrada válido — a regra "passa" por razões erradas.

### Checando o `Results.txt` remotamente

Se quiser ver o output textual do job sem abrir o dashboard completo, adicione `Results.txt` antes do `?` na URL do job:

```
https://prover.certora.com/output/<id>/<hash>/Results.txt?anonymousKey=...
```

## 9. URLs Especiais do Dashboard

Dado o link de output de um job:
```
https://prover.certora.com/output/<id_usuario>/<hash_job>/?anonymousKey=<chave>
```

Você pode derivar as seguintes URLs úteis:

### Status do job

```
https://prover.certora.com/jobStatus/<id_usuario>/<hash_job>/?anonymousKey=<chave>
```
Mostra os argumentos usados, a versão do Prover, e o status atual do job.

### Relatório legado com mais detalhes de erro

```
https://prover.certora.com/output/<id_usuario>/<hash_job>/FinalResults.html?anonymousKey=<chave>
```
Útil quando o relatório principal apresenta erro interno — pode conter mais informações.

### Output textual completo

```
https://prover.certora.com/output/<id_usuario>/<hash_job>/Results.txt?anonymousKey=<chave>
```
Captura o output de linha de comando da ferramenta.

### Análise de timeouts

```
https://prover.certora.com/output/<id_usuario>/<hash_job>/statsdata.json?anonymousKey=<chave>
```
Contém a seção `ANALYSIS` com o status de cada análise de bytecode. Um `false` em um método que frequentemente dá timeout pode indicar falha de análise — vale reportar para o suporte da Certora.

### Download completo (zip)

```
https://prover.certora.com/zipOutput/<id_usuario>/<hash_job>/?anonymousKey=<chave>
```
Baixa um arquivo `.tar` com todos os inputs e outputs do job. Requer estar autenticado no dashboard.

## 10. Diretório `.certora_internal`

A cada execução, o `certoraRun` cria (ou atualiza) o diretório `.certora_internal` no diretório atual.

### Arquivo mais útil

```
.certora_internal/latest/run.conf
```

Contém o `.conf` completo gerado automaticamente pela última execução. Use como base para criar seu próprio arquivo de configuração.

## 11. Uso em CI (GitHub Actions)

### Configuração básica

1. Adicione sua chave como secret no repositório com o nome `CERTORAKEY`.
2. Crie `.github/workflows/certora.yml`:

```yaml
name: certora-verification
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  workflow_dispatch:

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive

      - name: Install Python
        uses: actions/setup-python@v2
        with:
          python-version: "3.9"

      - name: Install Java
        uses: actions/setup-java@v1
        with:
          java-version: "21"
          java-package: jre

      - name: Install certora-cli
        # Pin a versão disponível em https://pypi.org/project/certora-cli/
        run: pip install certora-cli

      - name: Install solc
        run: |
          pip install solc-select
          solc-select install 0.8.23
          solc-select use 0.8.23

      - name: Run verification
        env:
          CERTORAKEY: ${{ secrets.CERTORAKEY }}
        run: certoraRun meu_projeto.conf
```

> **Nota:** Em CI, o comportamento padrão já é aguardar os resultados. O job de CI falhará se alguma propriedade for violada.

## 12. Erros Comuns e Como Investigar

### "Job submitted" mas sem resultado no terminal

**Causa:** `--wait_for_results` não foi usado.  
**Solução:** Adicione `--wait_for_results ALL` ao comando.

### Regras com TIMEOUT

**Sinais:** Símbolo de relógio laranja no dashboard, ou mensagem `Global timeout reached` no painel "Global Problems".

**Ações:**
- Reduza o escopo: rode apenas a regra problemática com `--rule`.
- Aumente o `--loop_iter` com cautela, ou use `--optimistic_loop`.
- Reduza concorrência: `--max_concurrent_rules 1` ou `2`.
- Verifique `statsdata.json` para ver se há análises falhando no método.
- Considere usar `--split_rules` para isolar regras pesadas em jobs dedicados.
- Use `--prover_args "-destructiveOptimizations enable"` como último recurso.

### Out of Memory

**Sinal:** Mensagem `Extremely low available memory` no painel "Global Problems" do dashboard.

**Ações:**
- Reduza `--max_concurrent_rules`.
- Simplifique summaries na spec.
- Verifique se há structs muito complexos sendo passados em funções.

### Regra sempre passa (suspeita de spec errada)

**Causa:** A regra pode ser vacuosa — as condições `require` eliminam todos os estados possíveis.  
**Solução:**
```bash
certoraRun ... --rule_sanity basic
```
Se a regra falhar no sanity check, a spec tem um problema lógico.

### Erros de compilação do Solidity

**Causa:** Versão do `solc` incorreta ou não disponível no `$PATH`.  
**Solução:**
```bash
# Verificar versão disponível
solc --version

# Especificar explicitamente na linha de comando
certoraRun Bank.sol --verify Bank:Bank.spec --solc solc0.8.23

# Ou no .conf
{ "solc": "solc0.8.23" }
```

### Opções depreciadas no .conf ou CLI

**Causa:** Versões recentes do Prover removeram algumas flags antigas.  
**Sinal:** Erro ao validar o `.conf` localmente (a validação agora ocorre antes de enviar).  
**Solução:** Consulte o [Changelog do Prover](https://docs.certora.com/en/latest/docs/prover/changelog/prover_changelog.html) para ver substituições.

Exemplo de migração:
```bash
# Antes (depreciado):
--prover_args "-adaptiveSolverConfig false"

# Agora:
--prover_args "-backendStrategy singlerace"
```
